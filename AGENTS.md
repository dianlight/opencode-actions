# AGENTS.md

## What this repo is

A shared library of GitHub Actions workflows that automate issue triage, PR review, and code implementation using the `anomalyco/opencode` GitHub Action. These workflows are synced to multiple downstream repos (see `.github/sync.yml`). This is NOT an application — it's infrastructure.

## Repository layout

```
.github/workflows/     # Workflows synced to other repos (the core deliverable)
scripts/               # Python maintenance script
config/                # YAML: task types, workflow→task mapping, fallback model scores
data/                  # Generated JSON (LiveBench scores, audit results) — do not edit manually
README.md              # Auto-generated by maintenance script — do not edit model tables manually
```

## Two automation pipelines

**Issue flow**: `/oc` comment on issue → `opencode-triage-issue.yaml` → (approval) → `opencode-new-pr.yaml` (creates PR + CHANGELOG, then dispatches tasks) → `opencode-implement.yaml` (per-task)

**PR flow**: PR opened/updated → `opencode-review.yaml` → `/oc` comment → `opencode-triage-pr.yaml` → (approval) → `opencode-implement.yaml`

Full diagrams in `.github/workflows/WORKFLOWS.md`.

## Deprecated workflows

`opencode-triage.yaml` and `opencode.yml` are no-ops. Do not reference or modify them.

## Key labels (state machine)

| Label | Meaning |
|-------|---------|
| `opencode:awaiting-response` | Waiting for human input |
| `opencode:approved-for-implementation` | Approved; ready to implement |

## Token strategy

- **Triage workflows** use OpenCode's GitHub App token (OIDC) so label changes trigger downstream workflow events.
- **Review and implement** use built-in `GITHUB_TOKEN` (`use_github_token: true`).
- `opencode-new-pr.yaml` uses `GITHUB_TOKEN` for API calls and OpenCode API key for CHANGELOG creation.

## Model management

- Models are scored against LiveBench leaderboards (`config/model-scores.yaml` has static fallbacks).
- `scripts/opencode_maintenance.py` fetches LiveBench data, audits workflow model usage, and regenerates README tables.
- **Free-first policy**: if a free model is within 5% of the best paid model, prefer the free one.
- Do NOT manually edit the "Model Recommendations" or "Workflow Model Audit" tables in README.md — they are overwritten by the maintenance script.

## Maintenance script

```bash
# Run locally (requires Python 3.14+ and pyyaml)
pip install pyyaml requests
python scripts/opencode_maintenance.py
```

Reads from `config/`, writes to `data/*.json` and `README.md`. Exits non-zero if any workflow uses a suboptimal model or if model coverage issues exist.

## Sync to other repos

`.github/sync.yml` maps which workflow files sync to which downstream repos. Uses `BetaHuhn/repo-file-sync-action`. Triggered manually via `sync-actions.yml`.

## Renovate

Auto-merges GitHub Actions version bumps weekly (Monday mornings).

## Maintaining this repo

### Adding a new workflow

1. Create `.github/workflows/<name>.yaml` in this repo (the source of truth).
2. Add it to `.github/sync.yml` under each downstream repo that should receive it.
3. Add an entry to `config/workflow-task-map.yaml` mapping the workflow name (without `.yaml`) to a task type from `config/task-types.yaml`.
4. Run the maintenance script to verify the new workflow is detected and scored: `python scripts/opencode_maintenance.py`.
5. The `opencode-maintenance` workflow also runs automatically every 7 days and creates a GitHub issue if any problems are found.

### Adding a new task type

1. Add the task type definition to `config/task-types.yaml` — include `name`, `description`, `signals` (keywords for auto-classification), `relevant_subscores` (LiveBench dimensions), `priority` (which subscore is most important), and optionally `requires_vision: true`.
2. Add workflow mappings in `config/workflow-task-map.yaml` to associate workflows with the new type.
3. Run the maintenance script. It will score models against the new subscores and update README.md tables.

### Adding a new model

1. If the model is on LiveBench, the maintenance script picks it up automatically from the leaderboard CSV.
2. If the model is NOT on LiveBench, add static scores to `config/model-scores.yaml` under `model_scores:` with keys: `coding`, `instruction_following`, `overall`, `reasoning`, `vision`.
3. Run the maintenance script to verify the model appears in README.md tables.

### Adding a new downstream repo

1. Add a new top-level key to `.github/sync.yml` with the `owner/repo` format.
2. List the workflow files to sync under that key.
3. The sync runs manually via `sync-actions.yml` (not automatic).

### Workflow model updates

When a new model tops the leaderboard or scores change:

1. Run `python scripts/opencode_maintenance.py` — it fetches fresh LiveBench data, scores all models, and updates `README.md` tables and `data/*.json`.
2. The script exits non-zero if any workflow uses a suboptimal model. To fix: update the `model:` field in the relevant workflow file to the recommended model from the audit table.
3. The script also exits non-zero on coverage issues (e.g., task types with no free model within threshold).

### Syncing workflows to downstream repos

- `.github/sync.yml` controls which files go where. Currently 4 downstream repos: `dianlight/smartmontools-sdk`, `dianlight/hasos_more_modules`, `dianlight/srat`, `dianlight/hassio-addons`.
- Uses `BetaHuhn/repo-file-sync-action`. Triggered manually via `sync-actions.yml`.
- Deprecated workflows (`opencode-triage.yaml`, `opencode.yml`) are still listed in `sync.yml` — they are no-ops in downstream repos.

## When editing workflows

- Workflow files in `.github/workflows/` are the source of truth for downstream repos.
- The `anomalyco/opencode/github` action reference is pinned by commit SHA.
- Triage workflows have a shell-based quick-approval pre-step that bypasses the AI for simple approvals.
- `opencode-review.yaml` skips draft PRs, PRs with 50+ changed files or 3000+ additions, and trivial-only diffs.
- All triage workflows filter out bot users (`github-actions[bot]`, `opencode-agent[bot]`, `coderabbitai[bot]`).
